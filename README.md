inpx-web-7z
===========

<a id="ключевые-отличия" />

## Что это за форк

Форк [`bookpauk/inpx-web`](https://github.com/bookpauk/inpx-web) для self-hosted библиотек с упором на большие `.7z` / `.zip` / `fLibrary` коллекции, Docker-развёртывание и обновлённую встроенную FB2-читалку. README ниже в первую очередь описывает сценарий `Docker / сервер`, но также покрывает standalone-бинарники и запуск из исходников через Node.js.

Ключевое в этом форке:

* запуск в Docker и Proxmox LXC, работа с большими `.7z` / `.zip` / `fLibrary` коллекциями и конвертацией через fb2cng, MuPDF и опционально Calibre
* персональные и внешние витрины: вкладки `Новинки`, `Популярное`, внешний источник и раздел `Для вас` с действиями `Обновить витрину`, `Неинтересно`, `Вернуть`
* обновлённый интерфейс каталога: карточки книг, серий и авторов, обложки, навигация по связанным сущностям и расширенный раздел `fb2 info`
* поддержка фото автора, краткой биографии из `fLibrary` и улучшенной работы с обложками и изображениями внутри `fb2`
* встроенная FB2-читалка с paged-режимом, мобильной адаптацией, debug-панелью и отдельным маршрутом `/#/reader-lab`
* отправка и конвертация книг в `epub`, `epub3`, `kepub`, `kfx`, `azw8` и `pdf`, а также отправка через Telegram и email
* многопользовательские профили, личные списки чтения, `Telegram chat id`, email и публикация через OPDS

## Какой сценарий выбрать

1. `Docker / сервер` — основной и рекомендуемый вариант для домашнего сервера, NAS, unRAID, Proxmox LXC, VPS и постоянного web-доступа к библиотеке.
2. `Standalone-бинарник` — локальный запуск без Docker, если нужен один готовый `.exe` / исполняемый файл.
3. `Запуск через Node.js` — режим для разработки, отладки и ручного старта из исходников.

Если нужен серверный каталог, переходите к разделам про Docker. Если нужен готовый исполняемый файл, используйте GitHub Releases. Если работаете с исходниками, смотрите раздел про Node.js ниже.

## Сборка Docker-контейнера

Для этого форка основной способ запуска рассчитан на Docker.

Готовые YAML-примеры:

* [docker-compose.full.yml](./docker-compose.full.yml) — основной образ с конвертацией через fb2cng и MuPDF
* [docker-compose.telegram-email.yml](./docker-compose.telegram-email.yml) — пример с отправкой книг в Telegram и на email

Важно: путь в `INPX` и каталог в `LIBDIR` должны совпадать с путём внутри контейнера из правой части `volume`.
Например, если библиотека смонтирована как `...:/library:ro`, то и `INPX` должен быть вида `/library/имя.inpx`, а `LIBDIR` — `/library`.

Собрать основной образ с лёгкой конвертацией через fb2cng и MuPDF можно так:

```sh
docker build -t inpx-web-7z:latest .
```

Полный образ с Calibre fallback для максимальной совместимости можно собрать отдельно:

```sh
docker build --target runtime-calibre -t inpx-web-7z:calibre .
```

Облегчённый вариант без конвертации в `epub` / `epub3` / `kepub` / `kfx` / `azw8` / `pdf`:

```sh
docker build --target runtime-lite -t inpx-web-7z:lite .
```

Если нужно пересобрать все слои заново без кэша:

```sh
docker build --no-cache -t inpx-web-7z:latest .
```

Важно:

* основной образ умеет `fb2 -> epub`, `epub3`, `kepub`, `kfx`, `azw8` через fb2cng и `fb2` / `epub -> pdf` через MuPDF
* `runtime-calibre` тяжелее и собирается дольше из-за установки Calibre, но оставлен как совместимый fallback для редких ручных экспериментов
* `runtime-lite` заметно легче и быстрее собирается, не умеет конвертацию книг и автоматически скрывает кнопки `EPUB` / `EPUB3` / `KEPUB` / `KFX` / `AZW8` / `PDF` в интерфейсе
* Dockerfile использует multi-stage сборку и удаляет клиентские npm-пакеты из runtime, чтобы не тянуть dev/UI-зависимости в контейнер
* при включённом BuildKit повторные сборки идут быстрее за счёт кэша `apt` и `npm`

После сборки контейнер можно запустить через `docker compose`:

```sh
docker compose up -d
```

или напрямую через `docker run` с тегом `inpx-web-7z:latest`.

### Обновление контейнера

Если вы обновили код или перешли на новый тег образа, удобнее всего пересоздать контейнер:

```sh
docker rm -f inpx-web 2>/dev/null || true
docker build -t inpx-web-7z:latest .

docker run -d \
  --name=inpx-web \
  --restart unless-stopped \
  -p 12380:12380 \
  -e INPX=/library/my-library.inpx \
  -e LIBDIR=/library \
  -e CACHE_DIR=/usr/local/bin/.inpx-web/cache \
  -v /mnt/user/appdata/inpx-web:/usr/local/bin/.inpx-web \
  -v /mnt/user/books/my-library:/library:ro \
  inpx-web-7z:latest
```

Если образ уже собран и нужно только перезапустить контейнер на новой версии, достаточно удалить старый контейнер и запустить ту же команду `docker run` снова.

### Быстрый старт через docker run

Пример запуска для библиотеки Flibusta / `fLibrary`:

```sh
docker run -d \
  --name=inpx-web \
  --restart unless-stopped \
  -p 12380:12380 \
  -e INPX=/library/my-library.inpx \
  -e LIBDIR=/library \
  -e CACHE_DIR=/usr/local/bin/.inpx-web/cache \
  -v /mnt/user/appdata/inpx-web:/usr/local/bin/.inpx-web \
  -v /mnt/user/books/my-library:/library:ro \
  inpx-web-7z:latest
```

Важно:

* в каталог `/library` должны быть смонтированы `.inpx` и файлы библиотеки, которые указаны в `INPX` и `LIBDIR`
* каталог `/usr/local/bin/.inpx-web` хранит базу поиска, кэш, конфигурацию и служебные данные, поэтому его лучше выносить в постоянный volume
* библиотека подключена как `:ro`, чтобы контейнер не изменял исходные архивы
* первый запуск может быть долгим: приложение создаёт поисковую базу, а конвертация в `epub` / `epub3` / `kepub` / `kfx` / `azw8` / `pdf` требует установленного в образе fb2cng и MuPDF
* после обновления образа контейнер удобнее пересоздавать: `docker rm -f inpx-web` и затем запускать команду снова
* веб-интерфейс после запуска будет доступен по адресу `http://<ваш_хост>:12380`

### Защита за reverse proxy

Если сервис публикуется через SWAG, Authelia/Authentik и CrowdSec, не открывайте порт `12380` напрямую в интернет. Лучше держать `inpx-web` во внутренней Docker-сети или публиковать порт только на localhost:

```sh
-p 127.0.0.1:12380:12380
```

Для дополнительной защиты от обхода proxy можно включить обязательную внешнюю авторизацию:

```sh
-e INPX_REQUIRE_AUTH=true \
-e INPX_AUTH_MODE=proxy \
-e INPX_TRUST_PROXY=true \
-e INPX_PROXY_AUTH_HEADER=Remote-User \
-e INPX_TRUSTED_PROXY_CIDRS=172.18.0.0/16
```

В этом режиме приложение отдаёт UI, API, WebSocket, OPDS, книги и обложки только если запрос пришёл от доверенного proxy и proxy передал заголовок авторизованного пользователя. Прямой заход из LAN на случайно открытый порт получит `403 Direct access is forbidden`. `/health`, `/ready` и `/api/index-status` по умолчанию остаются открытыми для Docker/Unraid health-check; если нужно закрыть и их, задайте `INPX_AUTH_EXEMPT_HEALTH=false`.

### Несколько INPX-источников

По умолчанию приложение работает как раньше: один `INPX` и один `LIBDIR`. Если нужно объединить несколько библиотек в один каталог, можно задать `INPX_LIBRARY_SOURCES`.

Формат через `docker run`:

```sh
-e INPX_LIBRARY_SOURCES='[
  {"id":"main","name":"Основная","inpx":"/library/main/main.inpx","libDir":"/library/main"},
  {"id":"archive","name":"Архив","inpx":"/library/archive/archive.inpx","libDir":"/library/archive"}
]'
```

Для такого режима нужно смонтировать все каталоги:

```sh
-v /mnt/user/books/main:/library/main:ro \
-v /mnt/user/books/archive:/library/archive:ro
```

Короткий формат тоже поддерживается:

```sh
-e INPX_LIBRARY_SOURCES='Основная|/library/main/main.inpx|/library/main;Архив|/library/archive/archive.inpx|/library/archive'
```

Важно:

* старые `INPX` и `LIBDIR` продолжают работать и автоматически превращаются во внутренний источник `main`
* каждая книга в базе получает свой `sourceId`, `sourceName` и `sourceLibDir`, поэтому скачивание и чтение берут файл из правильной библиотеки
* `.inpx` и книжные архивы могут быть как `.zip`, так и `.7z`; для `.7z` нужен 7-Zip CLI, для обычных `.zip` он не требуется
* при изменении любого `.inpx` из списка поисковая база будет пересобрана, потому что хэш базы считается по всем включённым источникам

### Кэш книг

Приложение хранит подготовленные книги в рабочем каталоге контейнера, внутри volume с данными приложения:

* gzip-кэш восстановленных книг
* временно распакованные `.raw` файлы
* результаты конвертации в `epub`, `epub3`, `kepub`, `kfx`, `azw8` и `pdf`

Этот кэш нужен для быстрого повторного скачивания, чтения и конвертации. Обычно вручную его чистить не нужно:

* сервер сам периодически подчищает кэш каталога книг
* лимит размера задаётся параметром `maxFilesDirSize`
* интервал проверки задаётся параметром `cacheCleanInterval`

Обложки кэшируются отдельно в `public-files/cover`. Лимит задаётся параметром `coverCacheSize`
в байтах или переменной окружения `INPX_COVER_CACHE_SIZE_MB` в мегабайтах. Кэш обложек чистится
автоматически вместе с книжным кэшем, интервал задаётся параметром `cacheCleanInterval`
(по умолчанию раз в неделю).

Если нужно очистить кэш вручную, удаляйте только каталог с книжным кэшем внутри appdata, не трогая весь каталог `/usr/local/bin/.inpx-web` целиком.

### Отправка книг в Telegram и на email

Форк умеет не только скачивать книгу, но и отправлять её:

* в Telegram через Bot API
* на email через SMTP

Кнопки `Telegram` и `Email` появляются в карточке книги только если интеграции включены и настроены. Настраивать их можно двумя способами:

* в интерфейсе: `Настройки` -> `Отправка книг`, доступно только администратору профилей
* через переменные окружения при запуске контейнера

В админ-панели можно сохранить параметры Telegram/SMTP и сразу проверить подключение: Telegram проверяется через Bot API `getMe`, SMTP — через подключение к серверу и `verify`.

Пример `docker run` с включённой отправкой:

```sh
docker run -d \
  --name=inpx-web \
  --restart unless-stopped \
  -p 12380:12380 \
  -e INPX=/library/my-library.inpx \
  -e LIBDIR=/library \
  -e CACHE_DIR=/usr/local/bin/.inpx-web/cache \
  -e TELEGRAM_BOT_TOKEN=<telegram_bot_token> \
  -e TELEGRAM_CHAT_ID=<telegram_chat_id> \
  -e TELEGRAM_CAPTION_TEMPLATE='${AUTHOR} - ${TITLE}' \
  -e SMTP_HOST=smtp.example.com \
  -e SMTP_PORT=587 \
  -e SMTP_SECURE=false \
  -e SMTP_USER=user@example.com \
  -e SMTP_PASS=<smtp_password> \
  -e EMAIL_FROM=user@example.com \
  -e EMAIL_TO=reader@example.com \
  -v /mnt/user/appdata/inpx-web:/usr/local/bin/.inpx-web \
  -v /mnt/user/books/my-library:/library:ro \
  inpx-web-7z:latest
```

Переменные для Telegram:

* `TELEGRAM_BOT_TOKEN` — токен бота от `@BotFather`
* `TELEGRAM_CHAT_ID` — чат или пользователь, куда отправлять книги
* `TELEGRAM_CAPTION_TEMPLATE` — подпись к книге, по умолчанию `${AUTHOR} - ${TITLE}`

Переменные для email:

* `SMTP_HOST` — SMTP-сервер
* `SMTP_PORT` — порт SMTP, например `587`
* `SMTP_SECURE` — `true` для SMTPS, иначе `false`
* `SMTP_USER` — логин SMTP
* `SMTP_PASS` — пароль SMTP
* `EMAIL_FROM` — адрес отправителя
* `EMAIL_TO` — адрес получателя книг

Важно:

* если Telegram не настроен, кнопка `Telegram` в интерфейсе скрыта
* если у пользователя не заполнен личный `Telegram chat id`, используется общий `TELEGRAM_CHAT_ID` из настроек
* обычный клик по `Telegram` отправляет исходный файл книги, а через выпадающее меню можно выбрать `epub`, `epub3`, `kepub`, `kfx`, `azw8` или `pdf`
* если SMTP не настроен, кнопка `Email` в интерфейсе скрыта
* если у пользователя не заполнен личный email, используется общий `EMAIL_TO` из настроек
* обычный клик по `Email` отправляет исходный файл книги, а через выпадающее меню можно выбрать `epub`, `epub3`, `kepub`, `kfx`, `azw8` или `pdf`
* в текущей реализации отправляется оригинальный файл книги
* после изменения переменных окружения контейнер нужно пересоздать; изменения из админ-панели сохраняются в `config.json`

### Пользователи, личные списки и OPDS

В системе есть отдельный профиль администратора. Только он может:

* создавать новые пользовательские профили
* удалять профили
* первично нарезать логины и пароли для пользователей

По умолчанию администратор создаётся автоматически с учётными данными:

* логин: `admin`
* пароль: `admin`

После первого входа эти данные лучше сразу поменять.

В форке есть профили пользователей. У каждого профиля свои:

* списки чтения
* адрес `email` для отправки книг
* `telegram chat id` для отправки книг
* настройка публикации подборок в `OPDS`
* собственные `логин` и `пароль` для входа в профиль

Как создать пользователя:

1. Откройте веб-интерфейс.
2. В верхней панели рядом с переключателем разделов выберите профиль `Администратор`.
3. Войдите с логином и паролем администратора.
4. Нажмите кнопку управления профилями.
5. Создайте новый профиль и задайте пользователю его `логин` и `пароль`.
6. При необходимости заполните `email`, `telegram chat id` и включите публикацию профиля в `OPDS`.
7. Сохраните профиль и передайте пользователю его логин и пароль.

Как работает пользователь:

1. Он выбирает свой профиль в верхней панели.
2. Вводит выданные ему логин и пароль.
3. После входа может сам менять свои `email`, `telegram chat id`, пароль и настройки публикации в `OPDS`.
4. Если нужно перейти в другую учётную запись, можно выйти из текущего профиля и войти в другой профиль прямо из окна `Профили пользователей`.

Как это работает с входом:

* администратор управляет созданием и удалением профилей
* обычный пользователь может редактировать только свой текущий профиль после входа
* полный список профилей и переключение между ними доступны только авторизованному администратору
* без входа защищённый профиль можно выбрать, но редактирование его настроек и личных списков будет недоступно
* пользователь может сам сменить свой пароль в настройках своего профиля
* администратор может сбросить пароль любого обычного профиля, но не может посмотреть старый пароль

Можно работать и без пользовательских учёток.

В таком режиме приложение остаётся полностью рабочим, но с ограничениями:

* нет разделения личных списков чтения между людьми
* нет персональных `Telegram chat id` и `email` для отправки книг
* нет защищённых профилей и индивидуальных паролей
* весь прогресс чтения, закладки и настройки читалки считаются общими для текущего открытого профиля

### Сброс администратора

Сброс логина и пароля администратора можно сделать без ручного редактирования базы профилей.

Через переменные окружения контейнера:

```sh
-e INPX_ADMIN_LOGIN=admin \
-e INPX_ADMIN_PASSWORD=strong-password \
-e INPX_RESET_ADMIN_PASSWORD=true \
```

Через CLI:

```sh
node server/index.js --admin-login=admin --admin-password=strong-password --reset-admin-password
```

Что это делает:

* гарантирует наличие системного профиля администратора
* при `INPX_RESET_ADMIN_PASSWORD=true` или `--reset-admin-password` принудительно обновляет логин и пароль администратора при старте

Важно:

* если меняете логин администратора через reset, он не должен совпадать с логином обычного пользователя
* после одноразового сброса флаг reset лучше убрать, чтобы пароль не перезаписывался на каждом старте

Как это работает дальше:

* отправка книги по кнопке `Telegram` идёт в `chat id` текущего профиля
* отправка книги по кнопке `Email` идёт на адрес текущего профиля
* списки чтения принадлежат только текущему профилю

У списков есть два режима:

* `Личный` — список виден только в веб-интерфейсе выбранного профиля
* `OPDS` — список дополнительно публикуется в OPDS для этого профиля

Если профиль не должен светиться в OPDS, у него можно отключить публикацию профиля целиком.

OPDS для конкретного профиля открывается так:

```text
http://<ваш_хост>:12380/opds?user=<profileId>
```

Где взять `profileId`:

* откройте управление профилями в веб-интерфейсе
* у каждого профиля там показывается его готовая OPDS-ссылка

Важно:

* старые списки автоматически мигрируют в дефолтный профиль `Основной`
* только списки с видимостью `OPDS` попадают в каталог читалок
* личные списки в OPDS не показываются
* логин профиля должен быть уникальным
* если хотите, чтобы пользователь настраивал профиль сам, обязательно задайте ему логин и пароль при создании

## Читалка и отладка FB2

Для отладки рендера и пагинации есть отдельный маршрут:

```text
/#/reader-lab
```

Что можно делать в этом режиме:

* открыть локальный `.fb2` прямо с компьютера
* загрузить sample-FB2 из подключённой библиотеки
* использовать ту же `Reader.vue`, что и основной маршрут `/#/reader?bookUid=...`
* проверять paged-режим и рендер без поиска книги в библиотеке

Полезные query-параметры:

* `debugReader=1` — включить debug-панель читалки
* `debugReadMode=paged` — сразу открыть режим страниц
* `debugReadMode=scroll` — сразу открыть режим ленты
* `sample=night-watch.fb2` — автоматически загрузить sample-файл

Пример:

```text
http://127.0.0.1:12380/#/reader-lab?sample=night-watch.fb2&debugReader=1&debugReadMode=paged
```

### Как работает раздел `Для вас`

`Для вас` строится только для авторизованного профиля. Если профиль не выбран или пользователь не вошёл в него,
персональные полки не показываются.

Сейчас раздел собирается из нескольких независимых полок:

* `Продолжить чтение` — книги с незавершённым прогрессом чтения
* `Из ваших списков` — книги, добавленные в личные списки чтения
* `Незаконченные серии` — следующий том серии, если предыдущие книги уже читались или отмечены прочитанными
* `Похоже на то, что вы читали` — рекомендации по авторам, сериям и жанрам из истории чтения и списков

Алгоритм похожих книг использует несколько сигналов:

* прочитанные и открытые книги добавляют вес своим авторам, сериям и жанрам
* книги, дочитанные почти до конца, влияют сильнее, чем просто открытые
* книги из личных списков тоже учитываются, особенно если они отмечены прочитанными
* недавние действия имеют больший вес, поэтому свежие интересы постепенно поднимаются выше старых
* уже прочитанные, добавленные в историю и скрытые через `Неинтересно` книги не предлагаются повторно

Чтобы витрина не превращалась в однообразную выдачу, применяется небольшая диверсификация:

* подряд не показывается слишком много книг одного автора или одной серии
* у карточек выводится причина попадания в полку: автор, серия, жанры, прогресс или список
* внутри одинаково подходящих вариантов есть стабильный дневной поворот выдачи: подборка может немного меняться
  от дня к дню, но не должна хаотично прыгать при каждом обновлении страницы

Полка `Похоже на то, что вы читали` по умолчанию загружает больше книг, чем остальные персональные полки.
Если подходящих вариантов больше, под ней появляется кнопка `Ещё рекомендации`: она добирает следующую порцию
по тем же весам, не меняя лимиты `Продолжить чтение`, `Из ваших списков` и `Незаконченные серии`.

Действие `Неинтересно` работает как отрицательный сигнал для персональных витрин: книга скрывается из `Для вас`,
а через полку скрытых рекомендаций её можно вернуть.

Что важно:

* название вкладки берётся из имени внешнего источника, а не зашивается в код
* если книга не найдена в локальной библиотеке, она всё равно показывается в витрине с пометкой, что её нет в библиотеке
* если внешний источник временно недоступен, используется последний успешный кеш витрины

## Standalone и ручная настройка

Этот блок нужен для локального запуска без Docker и ручной настройки. Здесь собраны готовый standalone-бинарник, запуск из исходников через Node.js и низкоуровневые справочные разделы: CLI, конфиг, режим удалённой библиотеки, фильтрация `.inpx`, reverse proxy и dev-режим.

Для серверного сценария по-прежнему удобнее использовать Docker-разделы выше.

Что важно для standalone-запуска:

* начиная с `1.6.2`, Windows standalone-архив включает утилиты для обложек (`7zr` и `djxl`), поэтому их не нужно ставить отдельно
* если библиотека лежит на сетевом или внешнем диске, а запись туда ограничена, лучше сразу указать `--data-dir`

### Сборка standalone-бинарников

<a id="native_run" />

### Локальный запуск через Node.js без Docker

Этот режим подходит для локальной отладки и ручного запуска из исходников без Docker.

Если вы используете уже готовый standalone-бинарник из GitHub Releases:

* распакуйте архив в отдельную папку
* положите рядом с ним `.inpx` и файлы библиотеки или укажите их через параметры
* при необходимости задайте отдельный каталог данных через `--data-dir`
* для Windows-архива `1.6.2+` дополнительные `7-Zip` и `libjxl` вручную ставить не нужно: нужные инструменты уже лежат рядом с `inpx-web.exe`

Примеры для готового Windows-бинарника:

```bat
inpx-web.exe
```

Если `.inpx` и библиотека лежат не рядом с `.exe`:

```bat
inpx-web.exe --lib-dir="Z:\Books" --inpx="Z:\Books\library.inpx"
```

Если библиотека находится на сетевом/внешнем диске и вы не хотите хранить рабочие данные рядом с ней:

```bat
inpx-web.exe --data-dir="%LOCALAPPDATA%\inpx-web" --lib-dir="Z:\Books" --inpx="Z:\Books\library.inpx"
```

Ниже пример для Ubuntu; для других Linux-систем различия минимальны:

```sh
# установка nodejs v16 и выше:
curl -s https://deb.nodesource.com/setup_16.x | sudo bash
sudo apt install nodejs -y

# подготовка
git clone https://github.com/AceAsket/inpx-web
cd inpx-web
npm i
npm run build:client && node build/prepkg.js linux

# удалим файл development-среды, чтобы запускался в production-режиме
rm ./server/config/application_env

# запуск inpx-web, тут же будет создан каталог .inpx-web
node server --app-dir=.inpx-web
```

### Конфигурация

По умолчанию, при первом запуске в рабочей директории будет создан конфигурационный файл `config.json`.
При необходимости, можно настроить нужный параметр в этом файле вручную. Параметры командной
строки имеют больший приоритет, чем настройки из `config.json`.

```js
{
    // рабочая директория приложения, аналог параметра командной строки --data-dir (или --app-dir)
    // пустая строка: использовать значение по умолчанию - <execDir>/.inpx-web
    // где execDir - директория файла приложения
    "dataDir": "",

    // директория для хранения временных файлов
    // пустая строка: использовать значение по умолчанию - <dataDir>/tmp
    // специальное значение "${OS}" указывается для использования системного каталога:
    // "${OS}" => "<os_temporary_dir>/inpx-web"
    "tempDir": "",

    // директория для хранения логов
    // пустая строка: использовать значение по умолчанию - <dataDir>/logs
    "logDir": "",

    // директория библиотеки (с zip- или 7z-архивами), аналог параметра командной строки --lib-dir
    // пустая строка: использовать значение по умолчанию - директорию файла приложения (execDir)
    "libDir": "",

    // путь к файлу .inpx, аналог параметра командной строки --inpx
    // пустая строка: использовать значение по умолчанию - inpx-файл, что найдется в директории библиотеки
    "inpx": "",

    // несколько INPX-источников; если массив пустой, используются старые libDir/inpx
    // каждый источник можно держать в отдельном volume или каталоге
    "librarySources": [
        {
            "id": "main",
            "name": "Основная",
            "inpx": "/library/main/main.inpx",
            "libDir": "/library/main",
            "enabled": true
        },
        {
            "id": "archive",
            "name": "Архив",
            "inpx": "/library/archive/archive.inpx",
            "libDir": "/library/archive",
            "enabled": true
        }
    ],

    // конфигурационный файл для фильтра по авторам и книгам (см. ниже)
    // пустая строка: использовать значение по умолчанию - файл filter.json в директории файла конфигурации
    "inpxFilterFile": "",

    // разрешить(true)/запретить(false) перезаписывать файл конфигурации, если появились новые параметры для настройки
    // файл перезаписывается с сохранением всех предыдущих настроек и с новыми по умолчанию
    // бывает полезно при выходе новых версий приложения
    "allowConfigRewrite": false,

    // разрешить(true)/запретить(false) использовать небезопасный фильтр (см. ниже)
    // аналог параметра командной строки --unsafe-filter
    "allowUnsafeFilter": false,

    // пароль для ограничения доступа к веб-интерфейсу сервера
    // пустое значение - доступ без ограничений
    "accessPassword": "",

    // таймаут автозавершения сессии доступа к веб-интерфейсу (если задан accessPassword),
    // при неактивности в течение указанного времени (в минутах), пароль будет запрошен заново
    // 0 - отключить таймаут, время доступа по паролю не ограничено
    "accessTimeout": 0,

    // включить(true)/выключить(false) возможность расширенного поиска (раздел "</>")
    // расширенный поиск не оптимизирован, поэтому может сильно нагружать сервер
    // чтобы ускорить поиск, увеличьте параметр dbCacheSize
    "extendedSearch": true,

    // содержимое кнопки-ссылки "(читать)", если не задано - кнопка "(читать)" не показывается
    // пример: "https://omnireader.ru/#/reader?url=${DOWNLOAD_LINK}"
    // на место ${DOWNLOAD_LINK} будет подставлена ссылка на скачивание файла книги
    // пример: "https://mydomain.ru/#/reader?url=http://127.0.0.1:8086${DOWNLOAD_URI}"
    // на место ${DOWNLOAD_URI} будут подставлены параметры (без имени хоста) из ссылки на скачивание файла книги
    "bookReadLink": "",

    // включить(true)/выключить(false) журналирование
    "loggingEnabled": true,

    // включить/выключить ежеминутный вывод в лог memUsage и loadAvg
    "logServerStats": false,

    // включить/выключить вывод в лог запросов и времени их выполнения
    "logQueries": false,

    // максимальный размер кеша каждой таблицы в БД, в блоках (требуется примерно 1-10Мб памяти на один блок)
    // если надо кешировать всю БД, можно поставить значение от 1000 и больше
    "dbCacheSize": 5,

    // максимальный размер в байтах директории закешированных файлов в <раб.дир>/public-files
    // чистка выполняется по cacheCleanInterval
    "maxFilesDirSize": 1073741824,

    // максимальный размер в байтах директории кеша обложек в <раб.дир>/public-files/cover
    "coverCacheSize": 536870912,
    
    // включить(true)/выключить(false) серверное кеширование запросов на диске и в памяти
    "queryCacheEnabled": true,

    // размер кеша запросов в оперативной памяти (количество)
    // 0 - отключить кеширование запросов в оперативной памяти
    "queryCacheMemSize": 50,

    // размер кеша запросов на диске (количество)
    // 0 - отключить кеширование запросов на диске
    "queryCacheDiskSize": 500,

    // периодичность чистки кеша запросов на сервере, в минутах
    // 0 - отключить чистку
    "cacheCleanInterval": 10080,

    // периодичность проверки изменений .inpx-файла, в минутах
    // если файл изменился, поисковая БД будет автоматически пересоздана
    // 0 - отключить проверку
    "inpxCheckInterval": 60,

    // включить(true)/выключить(false) режим работы с малым количеством физической памяти на машине
    // при включении этого режима, количество требуемой для создания БД памяти снижается примерно в 1.5-2 раза
    // во столько же раз увеличивается время создания
    "lowMemoryMode": false,

    // включить(true)/выключить(false) полную оптимизацию поисковой БД
    // ускоряет работу поиска, но увеличивает размер БД в 2-3 раза при импорте INPX
    "fullOptimization": false,

    // включить(true)/выключить(false) режим "Удаленная библиотека" (сервер)
    "allowRemoteLib": false,

    // включить(Object)/выключить(false) режим "Удаленная библиотека" (клиент)
    // подробнее см. раздел "Удаленная библиотека" ниже
    "remoteLib": false,

    // настройки веб-сервера
    // парамертр root указывает путь для кореневой страницы inpx-web
    // например для "root": "/library", веб-интерфейс будет доступен по адресу http://127.0.0.1:12380/library
    // root необходим при настройке reverse-proxy и встраивании inpx-web в уже существующий сервер
    "server": {
        "host": "0.0.0.0",
        "port": "12380",
        "root": ""
    },

    // настройки opds-сервера
    // user, password используются для Basic HTTP authentication
    // параметр root задает путь для доступа к opds-серверу
    "opds": {
        "enabled": true,
        "user": "",
        "password": "",
        "root": "/opds"
    },

    // страница для скачивания свежего релиза
    "latestReleaseLink": "https://github.com/AceAsket/inpx-web/releases/latest",

    // api для проверки новой версии, 
    // пустая строка - отключить проверку выхода новых версий
    "checkReleaseLink": "https://api.github.com/repos/AceAsket/inpx-web/releases/latest",

    // настройки по умолчанию для веб-интерфейса
    // устанавливаются при первой загрузке страницы в браузере
    // дальнейшие изменения настроек с помощью веб-интерфейса уже сохраняются в самом браузере
    "uiDefaults": {
        "limit": 20, // результатов на странице
        "downloadAsZip": false, // скачивать книги в виде zip-архива
        "showCounts": true, // показывать количество
        "showRates": true, // показывать оценки
        "showInfo": true, // показывать кнопку (инфо)
        "showGenres": true, // показывать жанры
        "showDates": false, // показывать даты поступления
        "showDeleted": false, // показывать удаленные
        "abCacheEnabled": true, // кешировать запросы
        "langDefault": "", // язык по умолчанию (например "ru,en")
        "showJson": false, // показывать JSON (в расширенном поиске)
        "showNewReleaseAvailable": true // уведомлять о выходе новой версии
    }
}
```

### Удаленная библиотека

В случае, когда необходимо физически разнести веб-интерфейс и библиотеку файлов на разные машины,
приложение может работать в режиме клиент-сервер: веб-интерфейс, поисковый движок и поисковая БД на одной машине (клиент),
а библиотека книг и .inpx-файл на другой (сервер).

Для этого необходимо развернуть два приложения, первое из которых будет клиентом для второго.

На сервере правим `config.json`:
```
    "accessPassword": "123456",
    "allowRemoteLib": true,
```

На клиенте:
```
    "remoteLib": {
    	"accessPassword": "123456",
        "url": "ws://server.host:12380"
    },
```

Если сервер работает по протоколу `http://`, то указываем протокол `ws://`, а для `https://` соответственно `wss://`.
Пароль не обязателен, но необходим в случае, если сервер тоже "смотрит" в интернет, для ограничения доступа к его веб-интерфейсу.
При указании `"remoteLib": {...}` настройки командной строки --inpx и --lib-dir игнорируются,
т.к. файлы .inpx-индекса и библиотеки используются удаленно.

### Фильтр по авторам и книгам

При создании поисковой БД, во время загрузки и парсинга .inpx-файла, имеется возможность
отфильтровать авторов и книги, задав определенные критерии. По умолчанию, для этого небходимо создать
в директории конфигурационного файла (там же, где `config.json`) файл `filter.json` следующего вида:
```json
{
  "info": {
    "collection": "Новое название коллекции",
    "version": "1.0.0"
  },
  "filter": "(r) => r.del == 0",
  "includeAuthors": ["Имя автора 1", "Имя автора 2"],
  "excludeAuthors": ["Имя автора"]
}
```
При фильтрации, авторы и их книги из `includeAuthors` будут оставлены, а из `excludeAuthors` исключены.
Использование совместно `includeAuthors` и `excludeAuthors` имеет мало смысла, поэтому для включения
определенных авторов можно использовать только `includeAuthors`:
```json
{
  "info": {
    "collection": "Новое название коллекции"
  },
  "includeAuthors": ["Имя автора 1", "Имя автора 2"]
}
```
Для исключения:
```json
{
  "info": {
    "collection": "Новое название коллекции"
  },
  "excludeAuthors": ["Имя автора 1", "Имя автора 2"]
}
```

Параметр `filter` используется для более гибкой фильтрации по атрибутам записей из .inpx.
Уберем все записи, помеченные как удаленные и исключим "Имя автора 1":
```json
{
  "info": {
    "collection": "Новое название коллекции"
  },
  "filter": "(inpxRec) => inpxRec.del == 0",
  "excludeAuthors": ["Имя автора 1"]
}
```
Использование `filter` небезопасно, т.к. позволяет выполнить произвольный js-код внутри программы,
поэтому запуск приложения в этом случае должен сопровождаться дополнительным параметром командной строки `--unsafe-filter`
или разрешением в конфиге `allowUnsafeFilter`.
Названия атрибутов inpxRec соответствуют названиям в нижнем регистре из структуры structure.info в .inpx-файле.
Файл `filter.json` можно расположить где угодно, что задается параметром `inpxFilterFile` в конфиге.

### Настройка https с помощью nginx
Проще всего настроить https с помощью certbot и проксирования в nginx (пример для debian-based linux):

```sh
#ставим nginx
sudo apt install nginx
```
```
#правим конфиг nginx
server {
  listen 80;
  server_name <имя сервера>;
  set $inpx_web http://127.0.0.1:12380;

  client_max_body_size 512m;
  proxy_read_timeout 1h;

  location / {
    proxy_pass $inpx_web;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
  }
}
```
```sh
#загружаем новый конфиг
sudo service nginx reload
```
Далее следовать инструкции установки https://certbot.eff.org/instructions?ws=nginx&os=debianbuster

### Proxmox LXC

Для Proxmox LXC в репозитории есть готовый установочный скрипт:

```sh
sudo ./scripts/proxmox-lxc-install.sh \
  --library-dir /mnt/books/flibusta \
  --inpx-file fb2.flibusta.lib.rus.ec.7z.inpx
```

Что делает скрипт:

* устанавливает Docker Engine и Docker Compose plugin внутри Debian/Ubuntu LXC
* копирует текущий репозиторий в рабочий каталог контейнера
* создаёт `docker-compose.proxmox.yml`
* собирает образ проекта и поднимает `inpx-web`

Перед запуском включите для LXC поддержку Docker:

```sh
pct set <CTID> -features nesting=1,keyctl=1
```

После этого контейнер нужно перезапустить.

Основные опции скрипта:

* `--app-root` — каталог, в который будет скопирован проект; по умолчанию `/opt/inpx-web`
* `--data-dir` — постоянный каталог данных; по умолчанию `/opt/inpx-web-data`
* `--port` — внешний HTTP-порт; по умолчанию `12380`
* `--lite` — собрать облегчённый образ `runtime-lite`
* `--skip-build` — не пересобирать образ, а только перезапустить compose-стек
* `--project-name` — имя docker compose проекта

Полный список параметров:

```sh
./scripts/proxmox-lxc-install.sh --help
```
